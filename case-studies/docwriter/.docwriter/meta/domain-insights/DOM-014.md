## Insight

The _api collection in Xperience docs follows a strict convention distinct from standard API reference documentation. Required elements: (1) frontmatter with only title/persona/identifier/order/redirect_from (no license/toc), (2) anchor toc / toc header navigation, (3) Dependency injection / Initialize required services section, (4) inpage_link 'toc' navigation between sections, (5) companion .cs files in the corresponding API examples directory with pageTitle, pageIdentifier, groupHeading, heading metadata tags. This convention is NOT documented in any guidelines file — it is purely a sibling-page convention discovered by analyzing contacts.md, contact-groups.md, form-data.md.

## Evidence

- GAP-003: T-002 and T-003 initially written as standard reference pages (prose, tables, admonitions) and required complete rewrites
- Sibling page analysis revealed the code-examples-only convention
- No formal invariant covers this convention
- Signal strength: high
- Confidence: low (first observation in a single run)

## Recommendation

Codify the _api collection convention as a formal invariant (candidate: INV-api-collection-convention). When creating new _api pages, the task planner MUST provide the structural template from sibling analysis.

## Applicability

Directly applicable to any future pipeline run creating pages in the _api collection. May generalize to other specialized collections with non-standard conventions.
