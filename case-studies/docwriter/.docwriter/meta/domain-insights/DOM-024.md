## Insight

Operations that share the same base ACL permission may have asymmetric additional checks at other layers. In the Xperience page permission model, both publish and unpublish require READ + UPDATE at the ACL layer, but publish additionally requires a workflow step role check (CanWorkWithContentItemInCurrentStep) while unpublish does NOT. This asymmetry is invisible when documenting permissions at the ACL layer alone — it only becomes apparent when examining the workflow layer independently.

When documenting operations that share base permissions, each operation must be independently verified for additional checks at every layer. Grouping operations by shared ACL permission and assuming uniform behavior leads to documentation errors.

## Evidence

- DOC-3167 T-002: Risk register explicitly flagged the publish/unpublish asymmetry: "the unpublish/revert distinction — workflow step check vs. no check — could be mischaracterized."
- DOC-3167 T-003: Workflows page (high technical accuracy risk) required careful per-operation documentation to avoid conflating publish and unpublish behavior despite their shared ACL permission.
- Strength: low (1 occurrence, single product-specific asymmetry)

## Applicability

Applicable to any permission documentation where multiple operations share a base permission but have different additional requirements at other layers (workflow, approval chain, feature flag). The task planner should instruct the content writer to document each operation independently, even when they appear grouped by shared permissions in code analysis output. Less applicable to simple permission models without layered checks.
