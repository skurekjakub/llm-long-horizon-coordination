## Insight

Xperience MemberRoleMemberInfo binding class has exactly 3 fields (ID, MemberID, MemberRoleID) with no temporal metadata. Any time-bound relationship (expiration, subscription periods) requires a separate custom object type with DateTime metadata columns. This architectural constraint shaped the entire DOC-3167 documentation task — the primary deliverable (T-001) architected a custom object type extension solution because the native binding class lacks temporal fields.

## Evidence

- DOC-3167: FIND-002 confirms MemberRoleMemberInfo has exactly 3 fields with no ExpiresAt/ValidTo. FIND-042 confirms the CMS_MemberRoleMember table has only 3 columns. All 18 tasks operated under this constraint.
- T-001 (the create task) was specifically designed to document the custom object type extension pattern as a workaround for this limitation.
- 3 related code findings (FIND-002, FIND-042, FIND-049) all confirm the constraint.

## Applicability

Applicable to future documentation tasks involving any Xperience binding class (Info class with only foreign-key columns). When the business requirement demands temporal or stateful semantics beyond what the binding class provides, documentation will need to cover the custom object type extension pattern. Generalizes beyond MemberRoleMemberInfo to any pure-FK binding class.
