## Insight

Xperience SecurityStamp regeneration is scoped exclusively to the MemberEnabled toggle — role assignment/removal does NOT trigger stamp regeneration. This creates a 0-to-30-minute session persistence window for role changes, where a member's cookie claims remain stale until the ASP.NET Core SecurityStampValidator revalidates (default: 30 minutes). Documentation touching role-based access must explicitly state what does NOT trigger revocation.

## Evidence

- DOC-3167: FIND-005 confirms MemberInfoProvider.Set() regenerates MemberSecurityStamp only when MemberEnabled changes. FIND-053 details the full session persistence chain.
- T-001 and T-002 both required accurate documentation of this subtle behavior.
- T-001's accuracy reviewer specifically flagged SecurityStamp claim verification as a verification target.

## Applicability

Applicable to any documentation task involving Xperience member authentication, session management, or role-based access control. The narrow trigger scope is a critical negative behavior that must be explicitly documented whenever role changes and session persistence are discussed.
