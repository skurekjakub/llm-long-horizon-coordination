## Insight

Xperience content-role bindings (ContentItemMemberRoleInfo) and member-role bindings (MemberRoleMemberInfo) are orthogonal cache domains. Deleting member-role bindings does NOT invalidate content-role caches, and this is correct behavior. Content access checks happen at the ClaimsPrincipal level via user.IsInRole(), not at the cache level. Documentation must present this cache independence as intentional design, not a bug.

## Evidence

- DOC-3167: FIND-022 confirms ContentItemMemberRoleRetriever caches depend on ContentItemMemberRoleInfo.All() and ContentItems.ById(), NOT on MemberRoleMemberInfo.
- FIND-052 synthesizes this as correct behavior: access checks happen at the ClaimsPrincipal level.
- T-006 and T-010 both needed to correctly document this non-invalidation as intentional.

## Applicability

Applicable to any documentation task involving Xperience content security, caching behavior, or the interaction between member roles and content access. When documenting cache invalidation scenarios, always verify which cache domain is affected.
