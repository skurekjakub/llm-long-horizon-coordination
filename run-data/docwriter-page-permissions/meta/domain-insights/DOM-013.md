## Insight

Xperience has two activity logging pipeline paths: (1) Standard Log — runs filters, then modifiers, then validators. Used for in-app tracking (page visits, landing pages, form submissions with full consent, custom activities). (2) LogWithoutModifiersAndFilters — runs only validators. Used for cross-site tracking, certain form submissions (legitimate interest), member registration. Cross-site tracking via CrossSiteActivityLogger bypasses modifiers entirely, creating a documentation coverage blind spot for modifier limitations. This dual-pipeline was the source of 3 accuracy rejections in the DOC-3137 run.

## Evidence

- T-005: misattributed email click to "Modifiers and validators" (actually validators-only)
- T-010: claimed activities are "saved to database" (actually queued in memory)
- T-011: used "standard activity logging pipeline" for an email click path that uses LogWithoutModifiersAndFilters
- Code analysis: ActivityLogService has 2 distinct pipeline paths
- Signal strength: high (4 tasks affected, systematic pattern)
- Confidence: low (first observation in a single run)

## Recommendation

Future documentation runs touching activity logging MUST include a pre-writing docFact with: (1) explicit lookup table of activity type → pipeline path, (2) which hooks run on each path, (3) persistence model (async memory queue, not sync DB write).

## Applicability

Directly applicable to any future documentation task involving Xperience activity logging, digital marketing, or cross-site tracking. The dual-pipeline pattern may also apply to other Xperience service architectures.
