## Insight

IActivityModifier, IActivityLogValidator, and IActivityLogFilter are NOT registered through standard DI/RegisterImplementation. They use ActivityEvaluationRegister.Instance (a custom singleton registry) with factory lambda registration called from Module.OnInit() methods. This is the only extensibility point in the analyzed Xperience codebase using this registration mechanism — all other service registrations use RegisterImplementation attributes or IServiceCollection DI. ActivityEvaluationRegister is in CMS.Activities.Internal (pubternal), making the registration mechanism itself unsupported for public documentation.

## Evidence

- Codebase map: ActivityEvaluationRegister.Instance.Modifiers.Register(() => new MyModifier())
- T-001 warning callout (lines 219-225) documents this unique pattern
- 4 internal modules use this pattern (code analysis AREA-003)
- Signal strength: medium
- Confidence: low (first observation in a single run)

## Recommendation

Future runs documenting Xperience extensibility points should check whether the registration mechanism matches the standard DI pattern or uses a unique one. Non-standard registration patterns need explicit documentation callouts and potential pubternal framing.

## Applicability

Directly applicable to Xperience activity logging documentation. Generalizable principle: check registration mechanism for any new extensibility point documentation.
