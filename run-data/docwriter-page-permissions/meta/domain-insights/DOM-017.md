## Insight

APIs with layered parameter architectures (parameter objects → query configuration delegates → cache settings) create documentation disambiguation challenges at each layer. When concepts at different layers share lexical roots (e.g., `IncludeUrlPath` on parameter objects vs `UrlPathColumns()` on query configuration), developers confuse them because the names are similar but the classes, inheritance chains, and behaviors differ.

ContentRetriever API exemplifies this pattern:
- **Layer 1 — Parameter objects**: `RetrieveParametersBase` with `IncludeUrlPath { get; init; } = true` (line 56)
- **Layer 2 — Query configuration**: `RetrievePagesQueryParametersBase` with `UrlPathColumns()` method (line 188) — page-only, absent from `RetrieveContentQueryParametersBase`
- **Layer 3 — Cache settings**: `ContentRetriever.cs` passes `parameters.IncludeUrlPath` to `GetCacheItemName` at 7 call sites (lines 105, 153, 278, 347, 402, 471, 522)

## Evidence

- DOC-2847 T-001: Required dedicated disambiguation section in content-retriever-api.md for IncludeUrlPath vs UrlPathColumns()
- DOC-2847 T-002: Required separate parameter table cells with cross-references between the two concepts
- DOC-2847 T-008: Cache key impact documented as a third dimension of IncludeUrlPath behavior

## Applicability

Applicable to any Xperience API (or external API) with 2+ configuration layers that share naming patterns. When documenting such APIs, explicitly map each configuration concept to its class, layer, and inheritance chain. Create dedicated disambiguation sections rather than relying on parameter tables alone.
