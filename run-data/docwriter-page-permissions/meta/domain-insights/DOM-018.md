## Insight

Documentation code blocks in `.md` files are not compiled, so non-existent API method calls can persist indefinitely until a systematic accuracy review discovers them. These phantom calls originate from:
1. API evolution: methods renamed or removed after docs were written
2. Copy-paste errors: method names from one type incorrectly applied to another
3. Wishful API design: documenting methods that were planned but never implemented

The `INV-codesamples-004` (build verification) safety net only works for compilable files (`.cs`, `.ts`). Documentation `.md` code blocks bypass this entirely, making accuracy review the sole defense.

Specific phantoms found in DOC-2847:
- `TopN()` on `RetrieveCurrentPageQueryParameters` — `RetrieveQueryParametersBase.cs` has no `TopN` symbol
- `OrderByDescending()` at 3 locations in reference docs — no `ContentRetriever` query parameter type defines this method; correct API is `OrderBy(OrderByColumn.Desc(...))`

## Evidence

- DOC-2847 T-001: TopN() phantom — fixed to Columns("ArticleTitle") which IS available on RetrieveQueryParametersBase
- DOC-2847 T-002: OrderByDescending() phantoms at lines 228, 816, 1057 in reference-content-retriever-api.md — required 2 accuracy review cycles to find all 3 instances
- Total: 6 phantom method calls across 2 files, 5 review cycles to fully resolve

## Applicability

Applicable to any documentation project with code examples embedded in Markdown files. Higher risk for large API reference pages (1000+ lines) where code blocks are scattered across multiple sections. When updating such pages, budget for accuracy review passes that verify every method call — not just newly-written content. Complements ANTI-010 (writer behavior) with the structural explanation (why the errors exist).
