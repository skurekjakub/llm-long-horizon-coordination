## Insight

Documentation pages reference code sample files via `code_link` tags, creating a dependency graph that spans multiple directory trees. When an API's calling convention changes (e.g., positional→named parameters), ALL call sites across ALL code sample directories must be audited — not just the ones co-located with the API reference docs.

In DOC-2847, the impact mapper found `Pages.cs` and `ContentItems.cs` (co-located in `APIExamples/`) but missed `ProductService.cs` (6 call sites, 5 section markers) and `ProductDataRetriever.cs` (1 call site, 3 section markers) in `DigitalCommerce/`. These files are referenced by `example-product-catalog.md` and `implementation.md` via `code_link` tags.

The gap hunter (GAP-002) caught the miss in verification cycle 1. The files were then addressed by T-009.

## Evidence

- DOC-2847 GAP-002: DigitalCommerce/ code samples missed by initial impact mapping, discovered by gap hunter
- DOC-2847 T-009: ProductService.cs (6 RetrieveContent/RetrieveContentByGuids calls) + ProductDataRetriever.cs (1 RetrieveContent call) converted to named parameters
- Verification matrix confirmed 6 code_link references across 2 doc pages pointing to digital commerce files
- All code_link section IDs remained intact after conversion (verified by cross-ref updater)

## Applicability

Applicable to any project using code_link (or equivalent include/reference) tags. When an API convention changes, the impact mapper should scan all code_link consumers across all directories, not just code files co-located with the API documentation. A code_link dependency scan during impact analysis would catch this proactively. Priority increases with the number of separate code sample directories in the repository.
