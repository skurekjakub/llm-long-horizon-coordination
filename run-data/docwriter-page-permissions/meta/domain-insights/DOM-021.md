## Insight

Permission systems with multiple interacting layers (e.g., application-level → ACL → workflow) require per-layer documentation sections with explicit documentation of inter-layer interaction rules. The Xperience page permission model exemplifies this:

- **Layer 1 — Application**: Channel admin bypass (skips ACL entirely)
- **Layer 2 — ACL**: WebPageAclPermissionEvaluator checks page-level permissions. Read is a prerequisite — granting Update/Create/Delete/Synchronize without Read has no effect (lines 118-123).
- **Layer 3 — Workflow**: Workflow step role checks apply to some operations (publish) but not others (unpublish), even when both share the same ACL permission (READ + UPDATE).

Without per-layer sections, the documentation conflates permission requirements from different enforcement points, leading to incomplete or misleading guidance.

## Evidence

- DOC-3167 T-001: Required 19 acceptance criteria and 2 initial review cycles to correctly document the 3-layer interaction model. The Read prerequisite rule spans layers (ACL enforcement, affects workflow operations).
- DOC-3167 T-003: Workflows page required careful disambiguation of which operations have workflow step checks vs which don't, despite sharing ACL permissions.
- The hub page (page-permission-management.md) organized content by operation rather than by layer, requiring cross-cutting callouts to document layer interactions.

## Applicability

Applicable to any product with 2+ permission enforcement layers that interact (RBAC + attribute-based, ACL + workflow, role-based + resource-based). When code analysis reveals multiple permission evaluation points in the call chain, the task planner should structure documentation to explicitly address each layer and their interactions. Less applicable to simple single-layer permission models.
