## Insight

Files modified by the pipeline may contain pre-existing encoding violations — Unicode en dashes (U+2013), curly quotes, non-ASCII whitespace — that the content writer cannot reasonably detect without tooling. These violations predate the pipeline run and are invisible in normal editing. When the pipeline modifies these files, the gap hunter detects the violations and attributes them to the pipeline, requiring a re-entry cycle to fix content the pipeline didn't introduce.

This is distinct from ANTI-002 (writer-introduced dash errors): the writer did not create these violations, and no amount of writer training or anti-pattern awareness prevents them.

## Evidence

- DOC-3167 GAP-002: Pre-existing Unicode en dashes (U+2013) found in files the pipeline was editing. T-003 converted 3 en dashes, T-006 converted 7 en dashes to double-hyphens. Hex-level grep confirmed zero remained after fix.
- ANTI-002 in the knowledge brief warned about writer-introduced dash issues, but this was a different failure mode (pre-existing content, not writer behavior).
- Both re-entry fixes passed first-attempt (style-only review).

## Recommendation

Add a pre-submission encoding validation step: after the content writer produces output, run a hex-level grep for common encoding violations (U+2013 en dash, U+2014 em dash, U+201C/U+201D curly quotes, U+00A0 non-breaking space) across ALL lines of modified files, not just the diff. This catches pre-existing violations before gap hunting and eliminates one category of re-entry cycles.

## Applicability

Applicable to any pipeline run that modifies existing documentation files. The risk scales with the age and number of contributors to the file — older files with many contributors are more likely to contain encoding inconsistencies. Less applicable when creating entirely new files.
