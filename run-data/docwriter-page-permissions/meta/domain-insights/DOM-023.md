## Insight

Operations that deviate from the standard permission checking pattern (e.g., READ + specific permission on the target page) require dedicated documentation callouts explaining the scope of the permission check. Three deviation patterns were identified in the Xperience page permission model:

1. **Ancestor-chain checks**: Synchronize checks READ + SYNCHRONIZE on the page AND all ancestors in the content tree. A single missing permission on any ancestor blocks the operation.
2. **Subtree evaluation**: Delete-with-children checks DELETE on all child pages with broken ACL inheritance (evaluateChildPages=true), not just the target page.
3. **Parent-page checks**: Create and Clone check permissions on the PARENT page, not the page being created/cloned.

Without dedicated callouts, users assume the standard "permission on this page" model applies to all operations, leading to permission configuration errors.

## Evidence

- DOC-3167 GAP-003: Synchronize ancestor-chain check was undocumented. Code analysis had documented this behavior (permissionSummary), but the task planner omitted synchronize from scope. Gap hunter caught it in cycle 1.
- DOC-3167 T-005: Delete subtree check was in risk register and documented correctly on first attempt, showing that when non-standard checks ARE in scope, they can be documented successfully.
- DOC-3167 T-004: Clone permission (READ on source + CREATE on target parent) was documented as part of GAP-001 resolution.

## Applicability

Applicable to any permission documentation task where code analysis reveals operations with recursive, ancestor-chain, subtree, or parent-context permission checks. The task planner should flag these operations as requiring dedicated callout sections, and the content writer should explicitly describe the scope of the permission check (which entities are checked, in what order, and what happens when one check fails).
